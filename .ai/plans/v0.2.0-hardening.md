# Plan: token-dealer v0.2.0 hardening

Generated: 2026-06-19 via `/grill-me` interview.
Owner: solo maintainer (passion project). Time budget: ~10–12 working days, part-time.
Scope: ship one tagged release that makes token-dealer suitable for solo self-deployment and open-source distribution, with friends as a secondary user surface on the maintainer's own instance (each friend gets their own API key).

This document is the authoritative plan. Future agents working on v0.2.0 should read this first, then check `git log --oneline` for in-flight work.

---

## Goal & non-goals

**In scope (v0.2.0):** 7-item floor covering config validation, secret management, rate limiting, two missing wire shapes, auto-migration from v0.1.x, docs, and CI hardening. Single-binary deployment (Docker or standalone). One semver release on the existing line.

**Explicit non-goals (deferred to v0.3+):** circuit breaker, metrics endpoint, `/v1/images`·`/v1/audio`·`/v1/video`, models.dev sync, per-agent rules engine, generic-adapter SSE, mid-stream budget enforcement, fuzz tests, signed releases, SBOM, web-based `/setup` wizard, Proxmox LXC support, CONTRIBUTING.md, horizontal scale / multi-region HA, SOC2-grade audit.

**Threat model:** untrusted internet clients can hit `/v1/*` (Bearer auth + rate limits + cost caps must hold); admin/UI surface stays loopback-only or behind a separate auth gateway.

## Floor items

### 1. Admin UI auth

- `[auth] enabled = true` becomes the **default**.
- On first start (no `token-dealer.toml`), the server **auto-generates a random admin password**, writes a minimal config, prints the password to logs **once** with a "save this, you will not see it again" banner, then starts.
- Admin rotates the password via `POST /admin/auth/rotate-password` (TBD exact path) once logged in.
- All `/ui/*` and `/admin/*` routes require auth when `[auth] enabled`.

### 2. API key encryption at rest

- **Algorithm:** AES-256-GCM (already in `Cargo.toml` via `aes-gcm`).
- **Scope:** only `[[auth.keys]] key = "..."` values in `token-dealer.toml`. Rest of config stays plaintext.
- **Master key source:** required env var `ROUTER_MASTER_KEY` (32 bytes, hex or base64 encoded).
- **KDF:** HKDF-SHA256 from the raw master key. No passphrase → no Argon2.
- **On-disk format:** `enc:<base64(nonce || ciphertext || tag)>`.
- **Failure mode:** refuse to start with a clear error if `[auth]` is enabled and `ROUTER_MASTER_KEY` is missing.
- **Rotation:** deferred. Document in SECURITY.md that rotation requires offline re-encryption (or a one-shot CLI subcommand in a later release).

### 3. Rate limiting

- **Algorithm:** token bucket, in-memory (no Redis).
- **Tiers:** per-API-key + global. Per-key 60 req/min burst 120; global 600 req/min burst 1200 (configurable).
- **Scope:** only `/v1/chat/completions`, `/v1/messages`, `/v1/responses`. `/v1/models`, `/health`, `/api/v1/events`, `/ui/*`, `/admin/*` exempt.
- **Counting:** request *start*, not completion (failed auth doesn't get a free retry).
- **Response on exceed:** `429` with `Retry-After: <seconds>` + OpenAI-shape JSON error envelope.
- **Config shape:** `[ratelimit]` section with global defaults + per-key overrides. **Per-key overrides must be obviously configurable** (top-level keys or a nested table — pick the one that scans best in the validator).
- **Escape hatch:** `enabled = false` for friend-scale deployments.

### 4. Pass-throughs: `/v1/messages` + `/v1/responses`

- **Pass-through only.** No cross-shape transpilation in v0.2.0.
- **Routing rule:**
  - `/v1/messages` inbound → upstream must speak `/v1/messages` (Anthropic, Kiro).
  - `/v1/responses` inbound → upstream must speak `/v1/responses` (OpenAI, OpenRouter Responses passthrough, etc.).
- **Tier routing still applies.** If the chosen tier primary is wire-incompatible, return `400` with a clear error: `"/v1/messages received but tier primary is openai/gpt-4o; use /v1/chat/completions or change tier primary"`.
- **`/v1/chat/completions`** stays the universal shape (full canonical transpilation, as today).
- **Streaming:** SSE chunks pass through unchanged on matching shapes.
- **Transpilation deferred to v0.3+** (Anthropic↔OpenAI, Responses↔ChatCompletions).

### 5. Config validation + first-run UX

**Validator:**
- Two-tier strictness after `serde` deserialization:
  - **Invalid fields** (wrong type, out-of-range, missing required, malformed `enc:` ciphertext) → **hard fail** with a path-pointing error (e.g. `[ratelimit.global].refill_per_minute: must be > 0`).
  - **Deprecated fields** → **warn-once** at startup, continue with new field's default.
  - **Unknown fields** → **warn-once** per field (forward-compat for users on newer configs running an older binary).
- **Implementation:** hand-rolled `validate(&Config) -> Result<(), Vec<ConfigError>>` pass after serde. No `validator` crate, no JSON-schema — surface area is small enough.
- **Hot-reload** (`/admin/config/reload`): same rules.

**`token-dealer check` subcommand:**
- `token-dealer check [--config path]`
- Reads config, runs validator, prints colored output.
- Exit codes: `0` = ok, `1` = invalid (will refuse to start), `2` = warnings only (will start).

**First-run / empty-config semantics:**
- Missing config file → server **auto-generates** a minimal `token-dealer.toml` (`[server]` from env/defaults, `[auth]` with generated admin password, empty `[[auth.keys]]`), prints admin password to logs once, starts.
- Existing config with empty sections (no providers, no tiers, no auth keys) → server starts, `/health` returns 200, `/ui/` works (dashboard shows "0 providers, 0 tiers"), but `/v1/chat/completions` returns `503` with `"no providers configured; visit /ui/providers to add one"`.
- **No `token-dealer init` subcommand in v0.2.0.** Can be added later if scripted deployments demand it. Web-based `/setup` wizard deferred to v0.3+.

### 6a. Auto-migration from v0.1.x

- On startup, server detects v0.1.x-shaped config via heuristic (presence of pre-v0.2.0 field layout, absence of `[ratelimit]`, plaintext `[[auth.keys]].key` with no `enc:` prefix).
- Action: copy `token-dealer.toml` → `token-dealer.toml.v0.1.bak`, rewrite in memory to v0.2.0 shape (add empty `[ratelimit]` with defaults, encrypt `[[auth.keys]].key` values **only if** `ROUTER_MASTER_KEY` is set — else leave plaintext + warn, add `[auth].admin_password` if missing), write to disk, one-line migration log.
- **Idempotent.** No interactive prompt (breaks non-TTY `docker compose up -d`). `token-dealer check` available for dry-run.

### 6b. Docs

- **README updates** (mandatory):
  - **Production checklist** — ~10 things to do before exposing to the internet: set `ROUTER_MASTER_KEY`, enable `[auth]`, set rate limits per person, set cost budgets, put behind reverse proxy, etc.
  - **Upgrading from v0.1.x** — points at the auto-migrate flow, mentions the `.bak` file.
  - **Security model** — what we promise (bearer auth on `/v1/*` when enabled, encrypted keys at rest, rate limits, cost budgets, SQLite audit log) and what we don't (zero-downtime, horizontal scale, nation-state defense, GDPR data-export/delete).
- **SECURITY.md** — responsible disclosure: `security@yourdomain`, 72h ack, fix timeline coordinated, public credit if desired. Supported versions: latest v0.2.x patched; v0.1.x EOL.
- **CHANGELOG.md** — Keep-a-Changelog format starting with v0.2.0. Sections: Added / Changed / Deprecated / Removed / Fixed / Security.
- **CONTRIBUTING.md** — deferred to v0.3.

### 7. CI improvements

Five targeted edits to `.github/workflows/build.yml`:

1. **Remove `continue-on-error: true`** from `cargo fmt --check` and `cargo clippy` so they actually block the workflow (today they print warnings but the workflow passes — broken signal).
2. **Add `cargo audit`** via `taiki-e/install-action` — catches known CVEs in the dep tree.
3. **`provenance: true`** on `docker/build-push-action` — SLSA-style supply chain metadata.
4. **Add a `release` job on tag push** — builds binaries for `linux/amd64`, `linux/arm64`, `macos/amd64`, `macos/arm64`, `windows/amd64`, uploads artifacts to the GitHub Release. Self-hosters who don't want Docker get a single binary.
5. **MSRV check as a separate non-blocking job** — `dtolnay/rust-toolchain@1.75` catches accidental MSRV breaks before they hit the stable job.

**Skip:** code coverage, fuzz testing, SBOM, signed releases.

## Open risks

- **Time estimate is rough.** Rate-limit + encryption + UI auth each have hidden complexity. Re-evaluate at end of Phase 1.
- **Auto-migration heuristic (6a)** might miss v0.1.x edge-case configs (custom providers, modified tier layouts). Need real-world v0.1.x configs to validate against before rc1.
- **`cargo audit`** may surface CVEs in transitive deps that require dependency upgrades — could expand scope. Triage before adopting as a blocker.
- **Web wizard deferred to v0.3.** Friends who can't be walked through `/ui/providers` manually may have a poor first-run experience. Acceptable risk per "functional for the most part" bar; revisit if friend-onboarding feedback is bad.
- **Per-key rate limit schema (item 3 refinement)** needs careful design to be discoverable — revisit during item 5 implementation.
- **Proxmox LXC deployment** deferred to v0.3+. Will need a fresh `/grill-me` pass when scheduled.
